## Architecture

Single Worker serves both API and static assets via dual endpoints:

| Endpoint | Domain | Auth | Purpose |
|----------|--------|------|---------|
| Browser | `bat.hexly.ai` | Cloudflare Access (Google OAuth) | Dashboard UI |
| Machine | `bat-ingest.worker.hexly.ai` | API Key | Probe data, monitoring |

```
Browser → Cloudflare Access → bat.hexly.ai → Worker → D1
                                              ↓
Probe → bat-ingest.worker.hexly.ai ──────────→ (same Worker)
```

## Build & Deploy

```bash
# Build UI into worker/static/
bun turbo build --filter=@bat/ui

# Deploy Worker (includes UI assets)
cd packages/worker && npx wrangler deploy --env production
```

Vite outputs to `packages/worker/static/`. Wrangler's `[assets]` config serves static files with SPA fallback.

## Local Development

```bash
cd packages/ui && bun run dev
```

- Access via `https://bat.dev.hexly.ai` (Caddy reverse proxy)
- `/api/*` proxied to production `bat-ingest.worker.hexly.ai`
- User info shows as anonymous (no Access JWT locally)

## Authentication Flow

1. Browser requests `bat.hexly.ai/*`
2. Cloudflare Access intercepts, redirects to Google OAuth
3. After login, Access sets `Cf-Access-Jwt-Assertion` cookie
4. Worker middleware verifies JWT signature via JWKS
5. Authenticated requests proceed to API/static assets

### User Info

`/api/me` extracts email from Access JWT:

```typescript
// Response
{ email: "user@gmail.com", name: "user", authenticated: true }
```

Sidebar displays email and initial avatar (Google profile photo not available via Access).
